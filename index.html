docker简单使用教程

2021-08-16

背景

最近实验室项目验收需要在多个验收地点部署演示环境，我们的演示环境包含一个老版本的rose编译器，在没有root权限的情况下，无法使用包管理，其依赖的安装十分繁琐，于是考虑预先打包docker镜像的方案。

本文主要写给验收地点的同学，他们完全没有接触过docker，所以是一个step-by-step的基础教程，有一定docker经验的同学可以不需要看本文，附录中整理了一组知乎上一个比较好的讲docker的文章，对我挺有帮助的，分享给大家。

docker镜像的导出

在我们对镜像做出一些自定义改动（如安装软件包等）后，需要通过执行一些命令来保存镜像，这里有两种方案

  1. 使用docker commit命令提交镜像，然后用docker save保存镜像，这种方式导出的镜像是分层的。
  2. 使用docker export命令导出指定容器，这种方式保存的镜像的是不分层的且没有入口信息。

理论上方式2导出的镜像会小一些，但是就失去了docker增量构建镜像的优势。根据使用场景不同选择不同的导出方式。

导入镜像

有两种导入导出方式不能混用，这一节的镜像使用docker save导出，所以使用docker load导入，另一种方式见下一节

导入测试镜像

  1. 下载得到docker-test.tar
  2. 运行docker load -i /path/to/docker-test.tar
  3. 屏幕输出一个image id如：Loaded image ID: sha256:f2a91732366c0332ccd7afd2a5c4ff2b9af81f549370f7a19acd460f87686bc7
  4. 运行docker run -it f2a91732366得到以下输出即为导入成功，可以继续进行下一步操作

导入镜像

  1. 下载得到rose-0.9.5a-withsrc-small.tar
  2. 运行docker load -i /path/to/rose-0.9.5a-withsrc-small.tar
  3. 这一步等待时间大概一分钟，同样得到一个image id如：Loaded image ID: sha256:6abfa470dc38da5dd727680811eeef94945faec58545933f1ade482f2c4b29b3
  4. 运行docker run -it 6abfa470dc38 /bin/bash 得到以下输出即为导入成功，可以继续进行下一步操作

使用容器

从用户的角度简单介绍一下容器技术，对于用户来说，容器表现得像一台Linux虚拟机，但是它的部署成本和性能相比虚拟机有很大的优势。镜像仓库提供大量的标准发行版环境和软件环境，让部署与维护变得很容易。它将用户空间虚拟化，提供了一套完全独立于宿主机的根文件系统与独立的运行时库环境，我们知道在Linux中切换系统软件的版本是一件很麻烦的事，需要处理许多依赖，同时很容易把系统的正常运行环境破坏，用容器就完美解决了这个问题。

回到我们的rose环境使用上，上面我们只用到了-it两个选项，这两个选项是打开交互所必须的选项，否则容器执行完后会直接结束。以上的运行方式docker与宿主机完全独立，文件不能相互访问。

我们用-v选项做一个目录的映射，冒号的左边宿主机目录，右边是容器中的目录。这样就能双向地传输文件。

目前在镜像中rose被安装在/opt/rose; rose的源码目录在/opt/rose-src; boost库安装在/opt/boost

8月16日更新

新增了带scp和mpi的镜像，采用了与之前不同的导出方式（两种镜像导出方式），需要用docker import命令导入而不是docker load命令。采用docker export导出的镜像（1928MB）会小于采用docker save导出的镜像(1991MB)，为方便网络传输采用了docker export导出。

学习路线

  1. 初次接触可以看一些比较好的博客，当初我看的是这篇博客还不错。经过这一阶段学习，可以达到正常使用Git的水平。
  2. 通过这个可视化Git学习网站，复习之前的知识同时强化对分支的理解。经过这一阶段的学习，可以熟练地管理分支，同时掌握一些实用的技巧。

初次使用

这一部分面向初次接触Git的同学介绍一下在Windows下安装和配置过程中值得注意的几点。

官网下载安装包

  • 换行符处理设置，默认是true建议选成input或false

  • commit用户信息设置

    • 首次运行git commit会提示设置用户信息，该信息不是认证信息，可以任意设置，但是考虑到代码的维护，最好留真实姓名与常用邮箱。

配置换行符处理

这一节解释为什么要改默认的 core.autocrlf 选项。

不同系统下的默认的换行符不同，Windows用两个字符CRLF表示换行，Linux用LF代示换行，跨平台开发时常常遇到换行符造成的苦恼。Git通过 core.autocrlf 选项控制换行符处理行为。

core.autocrlf取值 描述
true 签出时将LF转换为CRLF，commit时将CRLF转换为LF
input 签出时不做转换，commit时将CRLF转换为LF
false 签出和commit时均不做转换

不推荐将该设置设为true，因为主流的编辑器都能正确处理两种换行表示，而Linux下有的软件不能处理CRLF。在我们一个使用场景中需要将代码通过ssh拷贝到Linux服务器然后编译运行，设为true将导致编译出错。将autocrlf设为false人为保证所有的换行符都为LF是最为妥善的做法。

修改提交信息

场景：commit后发现commit信息错了或者是添加了不想commit的内容，但还没有push到远程仓库

有两个方法

  1. 通过 git commit --amend 重新输入commit信息
  2. git reset --soft {commid_id} 回滚到某一个commit，保留工作区和暂存区的内容，然后重新commit

方法1更简便，方法2可以回退多个版本。

C/C++对无参数函数声明的不同处理方式

2021-03-02

背景介绍

因为目标平台的原因，我所做的一个自动并行化项目是用C/C++混合编程的。由于一个新的需求，我需要为一个函数接口foo添加一个参数，当我添加完参数后，用gcc命令编译报重复声明。

很明显修改过后的代码不能通过gcc的编译，因为C语言不支持重载，两个foo是重复声明。那么问题来了为什么修改之前能通过编译呢？

测试

为了解答这个问题我们先构造一些小例子

按此规律构造出8个测试用例：

  • p0p1c0.c
  • p0p1c1.c
  • p1p0c0.c
  • p1p0c1.c
  • p1p2c1.c
  • p1p2c2.c
  • p2p1c1.c
  • p2p1c2.c
  • p2p0c2.c

可以看到，用gcc编译时能生成.o文件的是p1p0c1.c、p0p1c1.c、p2p0c2.c，用g++所有用例均能通过编译。

不难看出，没有不参数声明与有参数声明同时出现时无参数声明被忽略了。

真相大白

gcc在编译c语言时，对于重复的函数声明，在无参数与带参数声明同时出现的时候，不会产生冲突，这由于在c语言中，int foo()形式的声明表示foo可以有任意类型参数，编译器不做检查。与是无参数声明与带参数声明共同使用的结果为只有带参数的声明生效。这看起来像重载，实际上不是。

要想表达无参数，在c语言中需要声明为int foo(void)。而在c++中int foo()与int foo(void)是一样的。

总结

实际上原始的两个声明对于系统来说是一个bug，因为我们实际上想达到的目的是声明int foo(void)。这个bug由于我们在使用中一直使用的是带参数的接口所以一直没有暴露出来。

在调查的过程中，对于C语言对两个相同函数声明不报错感到很奇怪，甚至一度以为是编译器出bug了。虽然C++声称是C的超集，但是实际上总会有一些“阴暗的小角落”不经意的冒出来。

关于写博客的看法

2021-01-21

一切的开始

我是从什么时候开始有写博客的念头的呢？从大二开始身边的学长建议我写博客，因为他们说写博客对找工作很有帮助，我的内心告诉我不能这么功利，我打心底里抵触为了一个漂亮的简历而写博客这件事。

不服输的精神

时间来到大四，偶然间和朋友交流技术问题时得知身边不少同学已经有了自己的小站，写了不少的博文。自诩为技术达人(脸已打肿)，我不能在这方面落后，不就是搭建博客吗，以我多年的技术积累还不是分分钟搞定。

事实的确是这样，很短的时间内我搞定博客的部署，心里一阵窃喜。那时的我没有明白博客的价值是其中的博文而不是好看的外表。我是被动开始写博客，果不其然，在毕业后，我很快失去了打理博客的动力。

要强一直是我人生中的一个标签，不服输的精神促使我建立这个博客，但它不能支撑我爱上写博客。说到底，没坚持更新是性格使然。我是一个内向的人，热忠于接受输入而不乐与输出，空间、朋友圈年更是一个常态。更别说写博客这样公开展现自我的事情了。

态度的转变

时间转眼到了21年，经过研一上学期充实的学习生活，又经过了疫情的洗礼，转眼间我的研究生生涯已经过半，这一年有得有失，总的来说一切正向着好的方向发展。这一年我积累了不少，我都悉数记录在笔记里。

最近在做项目的过程中，偶然间我翻到了几个博客，我被博主们的精神打动了，我看了什么是互联网精神，什么是认真对待技术的精神。

榜样的力量

  • 刘未鹏
  • BYVoid

《C++0x漫谈》系列之：多线程内存模型-刘未鹏这篇文章给我眼前一亮的感觉，2007年那个未来的C++11标准还被称为C++0x的时代，在markdown还未普及的年代，该文章的排版如此规范，内容如此硬核。反观现在CSDN新的博文，转载泛滥，格式缺失，内容没有深度。

推薦一個神級輸入法——Rime
BYVoid的博客让我感到了一个生活态度，他是一个语言学爱好者，opencc的开发者，我推荐这篇文章是因为我通过这篇文章的推荐用上了rime输入法。博主旅居，环游全世界，他的博文让我感受到了一个人对生活的热爱，对爱好的专注。

当然打动我博主，促使我动笔的博主，远远不只上述两位。他们是两个时代的代表，前者是80后的老一代互联网人，后者是与我相同的90后新生一代。他们都很好地诠释了互联网精神，这不就是早已在我心中埋下种子的对技术、对生活的追求。

时间与空间

同时互联网是超越时间与空间的，2004年的文章与感悟，穿越16年与一为读者见面，这是一种神奇的体验。人的一生是需要留下些什么的，我觉得博客就是一个不错的选择。说不定赶上下一批GitHub北极代码保存计划。

自我驱动

事实上我对技术的是有热情的，我的第三篇的内容也早已想好，那是WHUT的毕业论文的Latex模板，是我和室友用Latex对学校的Word模版的像素级复刻。很可惜由于我懒，该文至今没有面世。究其原因还是上述的性格使然与被动的心态。我想这一点，在我发布本文的时候就不会再是我的桎梏了。

对自己的几点要求

  • 分享有价值的东西
    • 别人分享过的东西一定是自己消化过后的东西
    • 不出现纯转载，自身没有感悟的东西
  • 博文质比量重要，不追求数量，但求精致

如果说要做点什么的活，那就是现在开始写博客。

谨记，路遥知马力

emacs奇巧淫技

自从上次尝试了emacs，无意间发现了一个新的功能，那就是emacs内置了输入法！而且还不止一种语言

默认配置下按C-\就会启动或关闭输入法，M-x set-input-method便可以配置输入法，emacs内置了近百种输入法，能输入中文、日文、韩文等等世界各地的语言。很可惜其中没有五笔输入法，但是我就是想学五笔XD。

用内置输入法干点什么

用emacs输入中文可以干一些什么呢？这意味着可以在没有桌面的服务器环境上输入中文，另一方面在本地的字符终端中也可以通过这个方法实现中文的输入。

让我们尝试用字符终端输入中文

按Ctrl+Alt+F1进入tty1，这时你可以发现所有的中文字符都不能显示。这是因为终端只支持psf点阵字体，可以通过showconsolefont命令查看当前psf所有的字符，psf文件在/usr/share/consolefonts目录下。通过setfont可以换字体，但是一般情况下终端只支持8位编码，显然是不能支持显示所有中文字符的。

这里有两种方式，一种是给内核打中文补丁，一种是使用fbterm。就第一种方式来说，为什么发行版内核没有集成Unicode补丁。因为内核维护者是不会接受这样的补丁的，控制台只会在紧急情况下使用。一般情况下用户应该通过桌面环境或者fbterm之类的基于framebuffer的工具获得Unicode的支持。我们就采取第二种方式实现中文支持，从源里下载fbterm。在tty1中运行fbterm并指定字体与大小

然后就可以愉快地在emacs中输入中文了

如何优雅地使用终端进行调试emacs-gdb

2019-05-29

前言

用gdb调试的人，多半是被环境所逼。如果有条件用gui的调试环境为什么不用呢，难道用gui的调试环境会损失什么功能？难道在调用栈间进行跳转用鼠标点击不比输入up、down方便？

作为忠实的开源软件拥护者，我一度相信了网上所谓极客说的“抛弃gui吧！在gdb用命令行调试能让你更专注。“直到我对着命令行敲了无数遍list、无数遍bt。然后我开始寻找救赎，作为一个vim党我当然是先去找vim的gdb插件，发现vim对gdb的支持并不好，因为vim一开始并没有集成gdb的打算，仅有的几个插件已经很久没更新了。然后在知乎上看到这篇文章终端调试哪家强？对emacs的gdb插件有了一个好印象。但是作为坚定的vim党（逃）我怎么能去用emacs呢（其实是懒，不想学新的按键绑定），抱着试一试的心态用了一下emacs，最后。。。。真香

其实，如果对开源软件比较了解的话，你就会知道gdb和emacs都是出自Richard Stallman之手。那就不会去找vim的gdb插件了。

第一步

安装emacs，在各大发行版的仓库里都有，通过包管理安装即可。

第二步

Emacs自带gdb插件gud，无需手动安装。（是不是惊讶到了，早在2007年emacs就内置了这个模式emacswiki）

第三步

这一步需要学习emacs的基本操作（大概需要30分钟）

  1. 先记住C-x C-c（Ctrl-x Ctrl-c）（两个连续的组合键）为退出
  2. 运行emacs 按C-h t进入基础教走程跟着做一遍（中文教程）
  3. 补充几条教程里没有提到的命令
    • 多窗口的命令
      • C-x 1 只保留当前窗口
      • C-x 2 水平分屏
      • C-x 3 垂直分屏
      • C-x 0 关闭当前窗口
      • C-x o 切换窗口
    • 其他常用快捷键
      • C-x k关闭缓冲区

第四步

在~/.emacs启动脚本中加入(没有则创建)以下内容

第五步

输入终端中emacs -nw 打开 emacs 执行 M-x gdb [回车] 输入要调试的程序

每次都运行gdb命令很麻烦，在.bashrc中添加一个函数

#emacs gdb
gdbtool () { emacs -nw --eval "(gdb \"gdb -i=mi $*\")";}

直接输入gdbtool hello就能进入emacs调试模式了

这就是所谓的emacs的gdb插件，不是和命令行gdb一样吗？你需要看下一步

第六步

M-x gdb-ma[ny-window]括号中可以不输入，命令会自动补全

左 右
一般的gdb窗口 局部变量和寄存器
在gdb里用方向键上下回顾历史命令在这里变成了Alt-p和Alt-n
源代码 程序输出
可以用鼠标左键点击代码最左边设置和取消断点
调用栈 断点和线程
在调用栈和断点用鼠标中间键点击可以浏览源代码

赠品

  1. 窗口乱了怎么办？执行两次M-x gdb-ma[ny-window]
  2. 在用鼠标中间键点击浏览代码时，可能会触发gdb命令窗口被代码占据的bug，这时选择被占据的窗口用C-x k关闭buffer即可
  3. 我想临时把一个窗口放大，然后回到多窗口。用winner-mode。运行M-x winner-mode打开winner-mode，C-x 1关闭当前窗口以外窗口，然后通过C-c ←返回。
